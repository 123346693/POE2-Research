# 研究协议

这个项目的核心原则是：Build 推荐必须可追溯、可解释、可被版本更新推翻。

## 强度判断

“最强 Build”不只看最高伤害。默认评分会同时看：

- 伤害上限
- 刷图速度
- Boss 稳定性
- 防御层完整度
- 装备预算和关键装备稀缺度
- 是否适合 SSF
- 操作复杂度
- 版本稳定性与被削风险
- 数据可信度

当用户要求不同，权重也会变化。比如 HC 会显著提高生存权重，开荒会提高成本效率和 SSF 可行性，Bossing 会提高单体伤害权重。

## 数据分级

- A 级：官方补丁、游戏内数据、可复现测试、样本量足够的天梯/价格数据。
- B 级：POE2DB/Wiki 页面、公开 Build Guide、多个玩家交叉验证。
- C 级：单个作者观点、视频体感、未验证 Reddit/Discord 讨论。

推荐结果需要把数据可信度写进输出，避免把“热门”和“强”混为一谈。

## 工作流

1. 确定版本号和玩法目标。
2. 同步数据库和价格/天梯快照。
3. 生成候选 Build 池。
4. 按用户约束过滤明显不合适的候选。
5. 对剩余候选打分并解释。
6. 输出推荐、替代方案、风险和需要验证的数据。

## 不做的事

- 不用过期版本结论冒充当前版本答案。
- 不把单一视频标题当成 Build 强度证据。
- 不在缺少预算条件时默认推荐极高造价 Build。
- 不把软核交易服答案直接套给 HC/SSF 玩家。
